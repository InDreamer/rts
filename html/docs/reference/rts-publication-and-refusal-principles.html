RTS Publication and Refusal Principles#

This document distills the useful hard engineering constraints from the external review into current RTS language.

1. Runtime Projection Is A Release, Not A Cache#

The runtime view used by API, MCP, Q&A, agents, and pipelines is not a casual cache.

It is the service-facing version of governed truth. Therefore it needs a release record.

At minimum, each published projection should know:

  • source canonical revision
  • projection schema version
  • L0/L1 version
  • generated time
  • generated by
  • released by
  • activation state
  • previous/superseded version
  • rollback target

2. Do Not Silently Overwrite Latest#

RTS should not quietly replace the rules used by external callers.

When approved truth changes, create a new published version or artifact. Then explicitly activate it.

This lets RTS answer:

  • which truth version backed this answer?
  • which runtime projection was active then?
  • can we roll back if this release is wrong?
  • did two callers see different truth versions?

3. Pack-Level Release Is The Safe First Rule#

Phase one should treat a pack as the default release unit.

A rule, lookup, or helper from a pack should not enter the default service surface before the whole pack has passed review/signoff.

This avoids half-approved runtime truth.

Object-level release can be designed later, but it needs explicit governance.

4. Conflicts Must Fail Before Release#

RTS should not defer known conflicts to query time.

Conflict examples:

  • two objects use the same ID
  • two rules cover the same target without a precedence decision
  • two objects claim the same URI/address
  • COMMON and product-specific rules both appear applicable without an override rule

If these are unresolved, release should fail or remain blocked. The query layer should not guess which one is true.

5. Refusal Is A Correct Service Behavior#

RTS should refuse, degrade, or ask for clarification when truth state is not strong enough.

Examples:

  • scope is unresolved
  • active projection version is missing or not unique
  • candidate conflicts remain open
  • required dependency is not released
  • signoff/release granularity is insufficient
  • precedence is unclear
  • only L0/L1 summaries exist, but no original rule object is available

A refusal is better than a confident unsupported answer.

If RTS degrades instead of refusing, the degraded output must be explicitly labeled as bounded information-service continuity behavior. It must not masquerade as scenario judgment, approval, root-cause closure, QA signoff, or truth mutation.

6. L0/L1 Are Governed Navigation Outputs#

L0/L1 summaries are not decorative docs.

They affect what the system finds, what it skips, and what the LLM reads. That makes them part of runtime behavior.

They should have:

  • version
  • generation source
  • quality checks
  • regression examples
  • review or acceptance criteria appropriate to the risk

7. Deterministic Narrowing Comes First#

RTS should first narrow by known structure before semantic expansion.

Preferred order:

scope / metadata narrowing → L1 navigation → L2 object read → optional LLM explanation

Vector or semantic recall should not override scope, permission, approval state, or release state.

8. Dependency Hints Are Navigation, Not Truth Expansion#

Dependency hints can show related lookups, helpers, or rules.

They should help a reader continue from a known object to a related object.

In the first safe version, they should not:

  • widen search scope automatically
  • rank results ahead of scope and approval state
  • turn weak relationships into query evidence
  • make cross-pack expansion look like confirmed truth

9. Trace Must Connect Truth To Answer#

A useful trace is not only “what search found.”

RTS should connect:

canonical revision → projection/release artifact → query result → answer

This lets audit or debugging answer why a specific service response was trustworthy at that time.

10. Permissions And Retrieval Scope Must Match#

If a caller is not allowed to see a pack, rule, evidence, or review record, retrieval should not surface it.

Permission should align with retrieval scope, at least at pack level in early versions.

Later versions may add finer field-level or evidence-level permissions, but retrieval must never become a bypass around access control.

11. Ambiguity Reopen Requires A New Release Path#

If a previously closed ambiguity is reopened, it affects truth state.

That should create a new canonical revision and a new projection/release path.

Do not patch the current service answer in place without a traceable truth change.

12. Common Misimplementations To Avoid#

Avoid these mistakes:

  • mutable projection records replacing immutable release artifacts
  • object-level partial release by default
  • COMMON acting as an implicit fallback without a precedence rule
  • dependency hints becoming ranking features
  • canonical changes overwriting current projection in place
  • vector recall leading before scope/permission narrowing
  • query layer answering when conflict or dependency state is unresolved
