AI-first 知识产物不应该统一成一种文件格式

对 RTS 来说，正确问题不是“HTML、JSON、YAML、Markdown、CSV 哪个最好”，而是： 哪一层需要承载 truth、哪一层需要被 AI 审核和维护、哪一层需要检索召回、哪一层只是给 AI 的短上下文。 本报告按 AI/agent 直接阅读与审核约 80%，人工直接查看约 20% 的假设修正权重。 结论是分层组合：KB authoring 用受限 YAML/Markdown，runtime projection 用 JSON/JSONL，表格原始数据用 CSV/TSV 并投影成 JSON，HTML 只做 AI-mediated presentation，不做权威 truth。

8:2 推荐权重：AI/agent 阅读、审核、维护和运行时消费是主场；人类直接裸看只是少量抽查、裁决和审计场景。
JSON 当前 Java runtime store 已经把 projection manifest、object manifest、L2、governance summaries 等实现为 JSON/JSONL，并做 release、scope、hash、schema 校验。
Prompt 给 AI 的最终上下文不应等同于源文件。应由 resolver 从 projection 选择 L2/card/dependency，再渲染成短、可引用、可裁剪的上下文块。

Executive Recommendation

推荐基线：即使 KB 和 runtime projection 大部分时间都是给 AI 读，也不要让 AI 直接在仓库里自由读取各种后缀来判断 truth。AI 应该通过 RTS service 或 publisher 产出的受控 projection 读取；文件格式按层选择。人工可读性只作为副指标，不作为主导格式选择的理由。

AI-first 修正：这里的“可维护性”主要指 AI/agent 能稳定 diff、校验、重写、生成补丁和复核，不是让人长期裸眼阅读。HTML/Markdown 的价值也不是“给人看所以重要”，而是它们能作为 AI-mediated review 和 compact context 的渲染层。

格式评分矩阵

评分是针对“AI/agent 可安全使用、审核、维护 RTS rule truth”的综合权衡，不是通用文件格式排名。权重按 AI 80%、人工直接查看 20% 估算。5 分最好。

格式 Token 效率 可校验性 AI 维护/演进 运行时能力 AI 阅读效率 推荐角色
JSON 3 5 3 5 3 runtime L2、manifest、governance summary、API payload、hashable truth artifact。
JSONL 4 5 3 5 3 大量同构记录：object manifest、cards、edges、aliases、trace、index docs。
YAML 4 3 5 2 4 KB authoring、AI-editable canonical draft。发布前必须转成严格 projection。
Markdown 4 2 5 2 5 治理叙述、报告、AI prompt rendering、AI-mediated review，不承载最终结构化 truth。
CSV 5 3 3 4 2 大 lookup 表、mapping table、测试矩阵。必须配 schema/metadata。
TSV 5 3 3 4 3 比 CSV 更适合含逗号文本的表格，但仍需要 schema。
HTML 1 2 3 1 2 AI-mediated 展示、审阅、导航报告。不要作为 KB/projection 的权威输入。

为什么不是“全部给 AI 读 HTML”

HTML 在 AI-first 架构里的真实优点

  • 非常适合把 AI 分析结果组织成对比、层次、颜色、交互和长报告。
  • 可以把多个对象、依赖图、风险标记、样例放在一个 AI-mediated review 页面里。
  • 可作为审阅界面、交付物、training/解释材料，以及少量人工抽查入口。

HTML 不适合承载 truth

  • 标签、样式、脚本、导航元素会浪费 token。
  • DOM 结构不等于业务 schema，字段类型和必填约束弱。
  • 同一内容可被多种 HTML 表达，hash/canonicalization 麻烦。
  • AI 可能把装饰性文字、隐藏文本、注释或页面说明误读成事实。
  • 运行时服务很难基于 HTML 做稳定 scope、permission、dependency、L2 grounding。

推荐定位：HTML 是 presentation projection，不是 runtime projection。即使主要读者是 AI，也应该让 AI 读受控 context / L2 / card，而不是读完整 HTML DOM。HTML 可以由 JSON/YAML/Markdown 生成，但不要让 HTML 回流成 truth source。

分层处理方案

Layer 0
Source
原始证据和输入材料

PDF、Excel、XML、XSLT、Java、截图 OCR、业务说明等。这里格式由来源决定，不强行统一。RTS 只记录 source locator、hash、extraction state。

不直接作为默认 runtime truth 需要 evidence chain 可重抽取
Layer 1
KB Authoring
canonical rule / lookup / helper draft

主对象用受限 YAML：AI 易读、易 patch、易 diff，也便于少量人工抽查。治理说明用 Markdown，索引用 YAML/JSON。这里可以有注释和解释，但进入发布必须通过 schema validation。

YAML for objects Markdown for AI review/report exception human adjudication
Layer 2
Projection
approved truth 的服务运行视图

发布后生成 immutable JSON/JSONL projection：manifest、scope registry、object manifest、cards、dependency edges、content refs、L2 objects、governance summaries。这里是 query/index/agent 默认读取边界。

JSON / JSONL schema version release id content hash permission-aware
Layer 3
Index
召回和导航视图

从 projection 派生 L0/L1/card/search text/alias/confusable。它帮助 resolver 找对象，但不拥有 truth。命中后必须回读 L2。

JSONL Lucene/OpenSearch copy not truth owner
Layer 4
AI Context
模型实际看到的低噪声上下文

由 service 从 L2、dependency、authorized governance summary 生成。可以是 compact Markdown，也可以是 JSON envelope。关键是保留 URI、release id、hash、claim boundaries，而不是把整个文件塞给模型。

short snippets citations facts/inferences/unknowns prompt-injection safe
Layer 5
Presentation
AI-mediated presentation / HTML / PDF / dashboard

可展示 AI 生成的比较、推荐、图表、审阅视图，并支持少量人工抽查。它可以引用 projection 对象，但不应成为下一轮 runtime truth 输入。

面向 AI 的 token 策略

Token 效率不是简单看文件体积。对 RTS 更重要的是“单位 token 中有多少可 ground 的事实”。

策略 做法 原因
先定位，后读事实 用 scope + manifest + card + L0/L1 找候选，再读 L2。 避免把整个 KB 或 projection 塞进 prompt，减少 wrong-scope 污染。
prompt 渲染不要保留所有 JSON 噪声 给模型的 context 可渲染成紧凑 Markdown，但每段携带 URI/hash。 JSON 适合运行时，Markdown 更适合模型阅读；二者角色不同。
保留结构化语义，不保留治理长文 L2 保留 pipeline、inputs、target、dependencies、examples、warnings；raw evidence/review 长文只在授权时展开。 异常诊断、影响分析和测试规划需要结构化 L2，不需要默认读完整 review 历史。
表格不要转成自然语言 大 lookup 表保持 CSV/TSV 或数据库；prompt 中只给匹配行、key semantics、fallback behavior。 整表自然语言化会极大浪费 token，并削弱可校验性。
把候选、推断、事实分开 context 和 answer 都显式标注 facts / inferences / unknowns / candidates / human decisions。 这是 RTS authority boundary，不是表达风格。

具体格式建议

JSON / JSONL

用在：runtime projection、L2、manifest、cards、dependencies、trace、API/tool contract。

理由：它最适合服务端校验、hash、schema evolution、字段级读取、权限过滤和审计重放。

注意：不要把完整 JSON 原样全塞进 prompt。运行时用 JSON，AI 上下文可以渲染成更短的 blocks。

YAML

用在：KB authoring 的 rule/lookup/helper 主对象。

理由：AI/agent 修改体验好，diff 清楚，少量结构化逻辑可读性强；人工直接看只是少量抽查时的附加收益。

注意：YAML 有隐式类型、锚点、重复 key、解析器差异等风险。建议使用受限 YAML subset，并在 publish 前转 JSON canonical projection。

Markdown

用在：报告、review checklist、extraction notes、human decision narrative、AI context rendering。

理由：长文本最适合 Markdown；模型阅读、总结、批注和重写都自然。

注意：Markdown heading 和 bullet 不等于 schema。关键事实必须在 YAML/JSON 对象中有结构化来源。

CSV / TSV

用在：lookup table、mapping matrix、test case matrix、大量行数据。

理由：token 和存储都高效，适合表格工具处理。

注意：必须有 companion schema：delimiter、quote rules、column types、key columns、null/blank semantics、fallback behavior、source hash。

HTML

用在：报告、dashboard、对比页面、审阅 UI、AI-mediated explanation。

理由：能把结构、视觉层次和对比关系表达清楚，适合 AI 生成报告后给人少量抽查，也适合浏览器侧展示。

注意：HTML 不应被 query resolver 或 LLM 当 authoritative truth 输入。它应由 projection 生成，而不是替代 projection。

XML

用在：源系统样例、报文、schema/source evidence。

理由：银行报文和 transformation source 常见 XML/XPath。

注意：不要把 XML 原文直接当规则对象。规则应引用 XPath/source anchor，并在 L2 中保留结构化 source path。

决策树

如果你要保存的是... 优先选 不要优先选 发布前必须检查
一个 rule / helper 的结构化逻辑草稿 YAML HTML / 纯 MD object id、schema、target coverage、dependencies、examples、signoff state。
一个已发布 rule 的 runtime fact JSON YAML / MD release id、scope、L2 hash、content ref、permission visibility。
很多同类对象的列表 JSONL 单个巨型 JSON 每行独立 parse、release/type/schema 一致、可增量索引。
一个 lookup 表 CSV / TSV + schema 自然语言列表 key columns、output columns、blank semantics、fallback/reverse/coalesce。
AI review 讨论或 extraction report Markdown JSON 长字符串数组 把裁决、open questions、evidence refs 同步成结构化 governance summary。
给 LLM 的一次上下文 service-rendered compact context 直接塞目录里的原始文件 scope、URI、release、hash、truth/candidate labels、token budget。
给 AI / 少量人工抽查看的比较报告 HTML 把 HTML 当下一步 truth source 页面标明 derived from 哪个 release/projection，最好链接 URI/hash。

落地规范建议

应该做

  • 把 schema_version、release_id、scope、content_hash 作为 runtime projection 的硬字段。
  • 为 YAML authoring 定义受限 subset：禁止复杂 anchors、禁止重复 key、明确 boolean/date/string 规则。
  • 为 CSV/TSV 配 companion metadata，至少定义 delimiter、headers、types、key columns、blank semantics。
  • 让 AI 生成的 HTML/Markdown 报告引用对象 URI，而不是复制并改写 truth。
  • 让 LLM 通过 tool/API 读取 projection，不直接遍历文件系统。
  • 把 AI context 视为临时渲染产物，保存 pointer/hash/trace，不保存为新 truth。

避免做

  • 不要因为“AI 能读 HTML/Markdown”就用它们承载 L2 truth。
  • 不要让 OpenSearch/Lucene document 替代 L2 runtime object。
  • 不要把 YAML 注释、Markdown 叙述、chat history 或 trace feedback 写回 canonical truth。
  • 不要把 CSV 单元格类型、空值、fallback 交给模型猜。
  • 不要让 prompt 里的 release/scope/permission 文字代替 service gate。
  • 不要为了 token 少而裁掉 dependency、examples、warnings、source anchors 这些 grounding 必需字段。

最终推荐架构

source materials
  -> extracted assertions
  -> KB authoring
       rules/lookups/helpers: YAML
       evidence/review/reports: Markdown + YAML index
  -> AI-first validation/review + exception human adjudication/signoff
  -> immutable runtime projection
       release manifest: JSON
       scope/object/card/edge/navigation/trace: JSONL
       L2 runtime objects: JSON
       lookup raw rows: CSV/TSV + JSON metadata when needed
  -> RTS service gate
       release/scope/permission/hash/dependency/refusal/trace
  -> AI context renderer
       compact fact blocks with URI/release/hash
  -> managed answer / scenario report / HTML presentation

这套分层同时满足：AI 能读懂、服务能校验、工程可维护、运行时可索引、审计可复盘、truth boundary 不失守。

一句话结论

如果只能记住一条：权威 truth 用结构化、可校验、可 hash 的 JSON/JSONL projection；AI 阅读、审核和表达体验通过受控渲染解决；人工直接阅读只是少量抽查，不要反过来让阅读格式成为 truth source。
