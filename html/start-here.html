RTS Reading Experience

三分钟读懂 RTS

先不读长文档，用一条清楚的故事建立 RTS 的核心心智模型。

01

RTS 是规则真相服务，不是普通 RAG。

RTS 的核心职责是从多源材料形成 governed truth，再以 API、MCP、/ask 和 scenario endpoint 对外提供可追溯能力。

02

LLM 是核心能力，但不是 truth owner。

LLM 做受控分析、冲突发现、影响候选、测试候选和表达；事实必须回到 L2、dependency、授权治理视图或 trace。

03

candidate 边界不是能力上限。

场景服务可以做深入分析，但 impact/test/root cause 等输出必须区分 facts、inferences、unknowns、candidates 和 human decisions。

Mental Model

先记住这五句话。

  1. KB owns truth：完整治理图在 KB，而不是搜索索引或聊天记录。
  2. Projection serves truth：runtime projection 是 approved truth 的服务运行视图。
  3. Service gates truth：RTS service 负责 release、scope、permission、hash、grounding、refusal、trace。
  4. LLM analyzes truth：AI 可以分析和表达，但不能偷偷裁决事实。
  5. Human decides conflict：材料冲突和关键歧义最终由人工 review/adjudication 记录。

Reader Modes

按你的角色进入。

你要关心：答案是否有证据、scope 是否正确、unknown 是否被说清楚。

优先读项目总纲、Constitution 和 API guide。不要把 candidate 当 final business decision。

你要关心：projection、L2、dependency、trace 和 refusal 是否稳定。

优先读 projection contract、Day1 baseline、runbook 和 internal agent implementation plan。

你要关心：managed mode 与 tool mode 共享同一个 truth access boundary。

外部 agent 可以规划工作流，但每个 RTS tool call 仍要走 service gate。

你要关心：AI-first review 不是 AI-final decision。

AI 简化冲突、找证据、提问题；人工裁决和 signoff 才能进入 approved truth。
