RTS 项目介绍

项目概览

RTS 将分散的转换规则整理成结构化 KB，发布为 runtime projection，并借助内置 AI 分析，为服务、平台和外部 agent 提供可查询、可解释的规则能力。

解决什么问题

规则散落、难以证明、难以安全消费。

在报文转换场景里，同一条规则可能散在业务文档、Java/XSLT、Excel mapping、lookup table、样例和 review 记录里。RTS 将这些材料整理成 KB，发布为 runtime projection，并在同一服务边界内提供确定性查询和内置 LLM 分析。

现状 规则分散，靠人翻找和解释
建设 KB 沉淀规则、证据和 review
交付 Projection 支撑查询与内置分析
01

源材料 Source Material

业务文档、Java/XSLT、Excel mapping、lookup table、样例和澄清记录。

02

KB Knowledge Base

完整规则知识图：rule、lookup、helper、evidence、review、adjudication、signoff。

03

运行时投影 Runtime Projection

已批准内容的运行包：release manifest、scope registry、object manifest、cards、L2、dependency。

04

服务读取 Service Read

查询、索引、API、影响分析、内部 LLM 和外部 agent 都通过 projection 读取规则。

05

复盘与演进 Trace / Feedback

调用 trace、review feedback 和新裁决回到治理流程，再发布新 projection。

KB 与投影的分工

KB 保留完整知识上下文，runtime projection 提供运行时视图。

KB 不是给人看的长文档集合，projection 也不是简单摘要。两者都是机器优先结构，只是承担的职责不同。

KB 关注"规则为什么可信"

source、evidence、review、ambiguity、adjudication、signoff 和 lineage。

Projection 关注"服务如何读取"

release、scope、object manifest、L2、cards、dependency、governance summary 和 trace。

KB Layer

KB 里应该保留什么？

KB 是完整的规则知识图谱，保存规则本身，也保存规则为什么可信、哪里还有歧义、谁做了裁决。

  • rule / lookup / helper
  • source material
  • evidence chain
  • review notes
  • ambiguity / open question
  • human adjudication / signoff

Runtime Projection

Projection 里应该发布什么？

Projection 是已批准规则的服务运行视图。它降低运行时噪声，但保留完整的结构化规则事实。

  • release manifest
  • scope registry
  • object manifest
  • object cards
  • L2 runtime objects
  • dependency edges
  • governance summaries
  • caller profiles

为什么不直接用搜索索引？

索引能帮忙找到对象，但它本身不是规则事实。

Object card、L0/L1、alias、confusable 和全文索引适合召回和消歧。真正回答"规则怎么生成"时，服务需要回到 L2 runtime object、dependency 和授权 governance view。

先确定 scope 用 card 找对象 读取 L2 规则正文 展开 dependency 记录 trace

边界表

哪些内容负责治理，哪些内容负责运行？

内容主要用途介绍时怎么说
KB canonical objects保存完整规则和治理上下文这是规则知识资产本体
Runtime projection发布给服务读取的运行包这是已批准规则的服务视图
L2 runtime object保存服务可读的结构化规则事实回答事实时要回到这里
Object card / L0 / L1帮助搜索、导航和消歧它是入口，不是最终规则正文
Governance summary解释可信度、风险和 open question需要权限和场景控制
Trace复盘一次服务调用读了什么它证明过程，不反写规则真相

视角

不同场景的关注点。

价值是把规则资产沉淀下来，而非只做一个问答入口。

可以关注：规则来源能否追溯、发布版本是否清楚、后续能否支撑查询、审计、影响分析和自动化。

关键结构是 KB 到 runtime projection 的分层。

可以关注：manifest、scope、object manifest、L2 runtime object、object card、dependency、governance summary 和 trace。

内置 LLM 是核心服务能力，但规则权威留在 KB 和 projection。

内部 LLM 做内置分析，外部 agent 通过工具模式读取规则、依赖和 trace；两者都通过服务接口消费。

重点是证据、歧义和裁决能否留存。

KB 里要记录 source、evidence、review、open question、adjudication 和 signoff；projection 只发布运行时需要的视图。
