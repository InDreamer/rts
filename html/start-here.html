RTS 项目介绍

项目概览

RTS 将分散的转换规则整理成结构化 KB，发布为 runtime projection，并借助内置 AI 分析，为服务、平台和外部 agent 提供可查询、可解释的规则能力。

项目概览

RTS 解决的问题：规则散落、难以证明、难以安全消费。

在报文转换场景里，同一条规则可能散在业务文档、Java/XSLT、Excel mapping、lookup table、样例和 review 记录里。RTS 将这些材料整理成 KB，发布为 runtime projection，并在同一服务边界内提供确定性查询和内置 LLM 分析。

现状 规则分散，靠人翻找和解释
建设 KB 沉淀规则、证据和 review
交付 Projection 支撑查询与内置分析
01

KB 是规则知识的整理层。

它保存 rule、lookup、helper、source、evidence、review、ambiguity、adjudication 和 signoff。核心关注点：规则为什么可信。

02

Runtime projection 是运行时交付层。

它把已批准内容生成 release、scope、object manifest、L2 runtime object、dependency 和 governance summary，供服务稳定读取。

03

API、MCP 和 LLM 都是消费面，规则权威留在 KB 和 projection。

确定性查询、索引、内置分析和外部 agent 工具模式都基于 projection 读取规则，而非直接搜索草稿或未批准材料。

核心表述

五句话概括。

  1. RTS 建设的是双核心业务规则服务：稳定知识源和内置 LLM 分析，共同为服务、平台和外部 agent 提供规则能力。
  2. KB 保存完整上下文：规则逻辑、来源证据、review、歧义、裁决和 signoff。
  3. Runtime projection 把已批准内容发布成运行时服务可以读取的稳定包。
  4. 服务回答必须能回到 L2 规则对象、依赖关系、授权治理视图或 trace。
  5. LLM 是受控分析与表达层：可以解释、分析和提出候选，但不能替代规则事实或人工裁决。

视角

不同场景的关注点。

价值是把规则资产沉淀下来，而非只做一个问答入口。

可以关注：规则来源能否追溯、发布版本是否清楚、后续能否支撑查询、审计、影响分析和自动化。

关键结构是 KB 到 runtime projection 的分层。

可以关注：manifest、scope、object manifest、L2 runtime object、object card、dependency、governance summary 和 trace。

内置 LLM 是核心服务能力，但规则权威留在 KB 和 projection。

内部 LLM 做内置分析，外部 agent 通过工具模式读取规则、依赖和 trace；两者都通过服务接口消费。

重点是证据、歧义和裁决能否留存。

KB 里要记录 source、evidence、review、open question、adjudication 和 signoff；projection 只发布运行时需要的视图。
