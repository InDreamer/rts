RTS · Rule Truth Service & Runtime Projection

把分散的报文转换规则，整理成可发布、可追溯、可查询的业务知识服务。

报文转换背后是一组可复用的业务逻辑，散落在文档、代码、字段映射、查表和评审记录里。RTS 将它们收敛进结构化知识库（KB），再发布为运行时投影（runtime projection）。确定性查询、内置 LLM 分析、外部工具调用和业务平台，都从同一份投影获取一致的规则答案。

核心结构

两个核心：稳定知识源与内置 LLM 分析服务

稳定知识源

提供规则正文、来源证据、依赖关系、原文片段和审计追踪。

内置 LLM 分析

在服务内部完成规则解释、影响评估、测试建议和调查分析。

统一服务边界

知识库和运行投影共同支撑两个核心，分析能力始终基于已发布的规则内容。

工作边界

知识库承载已整理的业务规则；运行投影提供已批准内容的服务视图；RTS 统一管控发布版本、业务范围、调用权限、依据校验和追踪记录。

项目价值

核心价值：业务规则经结构化整理并借助 AI 分析，为服务、平台和 AGENT 提供可查询、可解释的规则能力。

RTS 将报文转换中的业务逻辑整理成结构化知识库，发布为运行时投影，并借助内置 LLM 分析能力，为服务、平台和外部 agent 提供确定性查询与智能解释。

业务逻辑可沉淀

规则从零散文档和代码中收敛，进入结构化知识库。

跨平台可复用

不同服务、agent 或业务平台基于同一份 KB/projection 获取一致答案。

运行边界可控制

服务统一管理版本发布、业务范围、调用权限、依据校验和审计追踪。

LLM/Agent 分析有依据

内置 LLM 和外部 agent 基于同一组原子能力做解释、影响评估、测试建议和调查分析。

项目主线

从材料，到结构化 KB，再到 runtime projection。

RTS 先把业务规则整理成 KB，再发布为 runtime projection，供查询、内置分析和外部 agent 统一消费。

核心过程

四步完成从材料到服务消费的完整链路。

规则先进入 KB 整理，再发布成 projection，最后由确定性查询、内置 LLM 和外部 agent 工具模式复用。

01

收集材料

业务文档、Java/XSLT、Excel mapping、lookup、样例和 review 结论。

02

整理成 KB

规则、lookup、helper、证据链、歧义、人工裁决和 signoff 被结构化保存。

03

发布投影

approved 内容被生成 release manifest、scope、object manifest、L2、dependency 等运行包。

04

服务消费

查询、API、搜索、内置 LLM 和外部 agent 都从同一份 projection 读取，并留下 trace。

服务消费模式

同一份规则投影，同时支撑内置 LLM 分析和外部工具调用。

RTS 将已批准的 KB 内容发布为 runtime projection 后，所有消费方都通过服务接口读取。内部场景由 RTS 内置分析引擎负责解释和表达；外部 agent 通过 REST/MCP 调用原子能力。两种入口共享版本、范围、权限和追踪机制。

内置分析

RTS 内部 /ask 和场景端点在服务边界内调用工具，生成规则解释、影响评估、测试建议和调查路径。

外部工具模式

Codex、CI bot 或业务平台通过 RTS API/MCP 读取规则、依赖、L2 和 trace，复用知识库能力但不承担规则权威。
