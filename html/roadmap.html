RTS 项目介绍

推进路径

RTS 的推进顺序是先锁定 governed truth 和 runtime projection 边界，再稳定查询、tool mode 和 managed analysis。

推进路径

先把规则真相和运行投影做扎实，再扩展服务和托管分析。

路线图从规则治理开始：先整理 KB，再发布 runtime projection，然后让确定性查询、外部 tool mode 和 managed LLM 场景稳定消费同一组原子能力。

KB 结构稳定 Projection 可发布 服务可查询和复盘
Step 1

整理规则材料进入 KB

把 rule、lookup、helper、source、evidence、review、open question 和 signoff 结构化保存。

Step 2

定义 runtime projection contract

明确发布包必须包含 release manifest、scope、object manifest、L2、cards、dependency 和 governance views。

Step 3

让服务只读取 projection

查询、API 和索引都从 active projection 读取，不直接读取 candidate pack 或工作区草稿。

Step 4

扩展可用场景

在稳定投影之上做规则查询、依赖解释、影响分析、测试建议、review 辅助和 pipeline 检查。

Step 5

稳定 managed analysis 场景

RTS 内部 LLM 在服务边界内调用受控工具做解释、冲突发现、影响候选、测试候选和调查分析；LLM 不可用时才降级为结构化信息服务。

判断标准

每一步真正要证明什么？

节点要证明的价值不要误解成
KB规则来源、证据、review 和裁决能被结构化追踪。一堆给人看的 Markdown。
Projection已批准规则能被发布成服务可读的稳定运行包。自然语言摘要或搜索索引。
Service查询能回到 scope、L2、dependency 和 trace。直接问模型或直接扫工作区文件。
Managed analysis在同一 truth boundary 内做解释、分析、影响候选和测试候选。让模型拥有 truth 或替代人工裁决。

表达重点

主线是规则如何从材料进入 KB，再如何发布成 projection，并通过同一个 RTS service 暴露为确定性原子能力、managed mode 和 external tool mode。

下一步真正重要什么

让 sample pack 到 runtime projection 的链路更真实、更稳定，再用 API、/ask 和场景端点把“查得到、查得准、可复盘、可受控分析”展示出来。

Glossary · 术语解释

推进路径里的关键英文词

这些词帮助同事理解项目从知识治理到服务运行的路径。Managed LLM 是核心服务能力之一，但不拥有规则真相。

RTS Rule Truth Source / Transformation Rule System

RTS 是双核心规则真相服务：既提供受控真相源原子能力，也提供 managed LLM 分析服务。

KB Knowledge Base，规则知识库

保存规则逻辑、来源证据、review 结论、歧义和 signoff 的完整治理层。

Rule Pack 规则包

按业务范围组织的一组 rule、lookup、helper，是整理、review 和发布的基本单位。

Runtime Projection 运行时投影

把 approved truth 发布成服务可读取的运行包，让查询、索引、managed analysis 和外部 tool mode 有稳定输入。

L2 Runtime Object L2 运行时规则对象

runtime projection 中最重要的事实正文，保存服务可读的结构化规则逻辑。

Object Card 对象卡片

用于定位、消歧和召回规则对象；它帮助找到 L2，但不能替代 L2。

Release Manifest 发布清单

说明当前 projection 来自哪里、包含什么、是否可用，以及出现问题时如何回滚。

Scope 业务范围

通常由 channel、product、pack、domain 组成，用来防止相似规则被串用。

Dependency 规则依赖

描述 rule、lookup、helper 之间的关系，是解释影响面和测试范围的基础。

Governance View 治理视图

在权限允许时展示 evidence、review、open question、risk 和 signoff 信息。
