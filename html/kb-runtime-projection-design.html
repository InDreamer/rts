One Rule's Journey

主链路只有一条，门禁挂在对应节点下面。

大节点表示规则产物如何前进：source 进入 KB，KB 发布 projection，service 读取 projection。每个大节点下方的小卡片说明它为什么能继续往下走，或服务到底读了什么。

治理门禁

规则进入 KB 后，需要有来源证据、review 结论、demo signoff 或人工裁决记录，说明这条规则为什么可信。

发布门禁

KB Pack 通过 schema、依赖完整性、content hash 和 release 条件检查后，才能生成 runtime projection。

导航读取

Runtime projection 里的 object card、L0、L1 帮助定位候选对象，适合召回和消歧，事实回答需要进一步读取 L2。

事实读取

RTS Service 回答事实时读取 L2 runtime object、dependency 和 trace。没有这些落点时，正确行为是降级、拒答或要求澄清。

KB Pack Structure

KB pack 将规则逻辑与可信依据分层组织。

这个 FXD.NDF pack 把可执行的规则逻辑放在 rules、lookups、helpers 里；把来源证据、review 结论和检查报告放在 evidence、review、reports 里。读规则时不被历史讨论淹没，需要追溯时又能找到依据。

规则逻辑：系统实际怎么转换

rules/ 保存 6 条 target rule，lookups/ 保存 cutoff lookup，helpers/ 保存 quoted currency pair 的复用逻辑。

Scope 是安全边界，不只是筛选条件。

这个 pack 的 scope 是 Tradition -> Stella / FXD.NDF / cutoff-fixing。相似字段可能出现在 FXO 或其他 product 里，scope 先收窄范围，才能避免相似但不适用的规则被错误引用。

Real Objects

用三类真实对象看 KB 的颗粒度。

这一部分展示 rule、lookup、helper 的真实逻辑和例子，是规则包从概念进入项目实物的关键。

Rule

primary rate source 规则

rule_fxd_ndf_primary_rate_source 读取 fixing currency pair 和 cutoff code，通过 shared cutoff lookup 得到 RTRS 和 TAIFX1。

查看规则片段 
inputs:
  fixing_currency1: USD
  fixing_currency2: TWD
  cutoff_code: TW
logic:
  lookup: lk_fxd_ndf_cutoff_by_pair_and_locode
  return_field:
    - hedge_primary_source
    - hedge_primary_source_page
target:
  fixing/primaryRateSource/rateSource
  fixing/primaryRateSource/rateSourcePage
result:
  RTRS / TAIFX1
Lookup

cutoff lookup

lk_fxd_ndf_cutoff_by_pair_and_locode 先用 forward pair 查 USDTWD|TW，必要时再 reverse fallback。

查看 lookup 逻辑 
pipeline:
  concat(ccy1, ccy2) -> forward_pair
  compose("{forward_pair}|{cutoff_code}")
  lookup(forward_key) -> forward_result
  concat(ccy2, ccy1) -> reverse_pair
  lookup(reverse_key) -> reverse_result
  coalesce(forward_result, reverse_result)

sample:
  USD,TWD,TW -> USDTWD|TW -> forward hit
  TWD,USD,TW -> USDTWD|TW -> reverse fallback
Helper

quoted currency pair helper

hlp_fxd_ndf_fixing_quoted_currency_pair 判断 quoteBasis 是否等于 source pair，等于时交换 currency 并输出 Currency1PerCurrency2。

查看 helper 逻辑 
source_currency1: USD
source_currency2: TWD
source_quote_basis: USDTWD

source_pair = concat(USD, TWD)
is_inverse_quote = (quote_basis == source_pair)

if true:
  target_currency1 = TWD
  target_currency2 = USD
  target_quote_basis = Currency1PerCurrency2

Runtime Projection

Projection 是”已批准规则的服务运行包”。

当前 sample runtime store 的 active release 指向 rel-2026-05-06-photo-fxd-ndf-cutoff。服务从这个 release 的 manifest、scope、object manifest、navigation、L2、dependency 和 governance view 里读取，而非直接读 KB 草稿。

当前服务指针 
{
  "active_release_id": "rel-2026-05-06-photo-fxd-ndf-cutoff",
  "rollback_target_release_id": "rel-2026-05-06",
  "updated_at": "2026-05-07T00:00:00Z"
}
当前 scope 
{
  "channel": "tradition",
  "product": "stella",
  "pack": "fxd-ndf-cutoff-fixing",
  "domain": "cutoff-fixing",
  "permission_boundary": "demo-signoff-photo"
}
运行包区域 它在产品上回答什么 边界
active-release.json现在服务运行哪一版？是指针，不是规则事实。
release-manifest.json这版从哪里来，能不能用？manifest 无效就没有 runtime truth。
scopes.jsonl问题属于哪个业务范围？scope 先行，不能全文搜索先行。
object-manifest.jsonl有哪些 released 对象？确定性查找优先从这里开始。
navigation/不知道精确 id 时怎么找到对象？card / L0 / L1 是入口，不是最终事实。
l2/规则到底怎么做？事实回答必须回到 L2。
dependencies/rule 依赖哪个 lookup/helper？用于解释、影响分析和测试规划。
governance/为什么可信？哪里有风险？按权限展开，不混进默认 operational answer。
traces/这次回答读了什么？用于复盘，不反写 truth。

L0 / L1 / L2

同一个规则对象，会被投影成不同精度的读取视图。

L0/L1 是为了低成本找到对象和消歧，L2 才是服务回答事实时必须读取的结构化规则正文。

L0

极短候选摘要

帮助快速判断“这个对象可能相关”。

primary rate source
FXD.NDF fixing
uses cutoff lookup
L1

结构化导航卡片

帮助确认对象、scope、target、依赖和风险。

object_id: rule_fxd_ndf_primary_rate_source
target_path: fxd.ndf.fixing.primaryRateSource
dependencies:
  - lk_fxd_ndf_cutoff_by_pair_and_locode
state: released
L2

结构化事实正文

保存 inputs、logic pipeline、target binding、examples、content hash。

logic:
  read currency1/currency2/cutoff_code
  lookup hedge_primary_source
  lookup hedge_primary_source_page
example:
  USD,TWD,TW -> RTRS / TAIFX1

关键约束

如果只有 L0/L1/card 命中而没有 L2 object，查询层应拒答或降级为”只能定位候选，无法给出规则事实”。这正是 RTS 比普通搜索更可靠的地方。

Query Walkthrough

一次真实查询应该怎样穿过 RTS？

场景：用户问 Tradition 到 Stella 的 FXD.NDF 交易，USDTWD 的 primary rate source 是什么？

Scope Resolution

先把问题收敛到 tradition / stella / fxd-ndf-cutoff-fixing / cutoff-fixing。scope 不清时，宁可要求澄清。

Permission Check

检查 caller 是否允许访问这个 pack、entrypoint 和输出模式。权限控制的是规则访问边界。

Object Discovery

优先从 object-manifest.jsonl 和 object card 找到 rule_fxd_ndf_primary_rate_source，而不是直接全文搜索。

L2 Read

读取 rule 的 L2：它说明输入是 currency1/currency2/cutoff_code，逻辑是调用 shared cutoff lookup，并绑定到 primary rate source 目标字段。

Dependency Load

加载 lookup L2：lk_fxd_ndf_cutoff_by_pair_and_locode。对于 USD,TWD,TW，命中 USDTWD|TW。

Grounded Answer

返回事实：hedge_primary_source = RTRS，hedge_primary_source_page = TAIFX1。同时引用 rule 和 lookup 作为依据。

Trace Emit

记录 query id、release、scope、selected URI、L2 read URI、dependency、grounding 和必要的 refusal/degraded reason。

Failure Is A Feature

RTS 的可靠性，很大一部分来自“该拒答时拒答”。

Scope 不清

不能确定 product/pack/domain 时，应要求澄清，而非拿相似规则猜测。

L2 不可读

只找到 card 或 L1 时，只能说”定位到候选”，事实回答需要 L2 支撑。

依赖未发布

rule 找到但 lookup/helper 不在同一 release 边界里，应降级或拒答。

治理仍有冲突

open question 或 precedence 没有裁决时，运行时不应临时修复。

一句话原则

Unknown is safer than wrong. 对 RTS 来说，拒答、降级和要求澄清是保护机制，防止把错误规则或无根据分析包装成事实。
