RTS · Rule Knowledge Base & Runtime Projection

把分散的报文转换业务规则，沉淀为可发布、可查询、可复用的 KB。

报文转换背后是一组可复用的业务规则逻辑，不只是字段搬运。RTS 将散落在文档、代码、mapping、lookup 和 review 记录里的规则整理成 governed KB，并发布为 runtime projection。服务、内部 LLM、外部 agent 和其他业务平台都可以在同一服务边界内查询、解释、分析和复用。

工作边界

KB 承载可治理的业务规则逻辑；runtime projection 提供运行时视图；RTS service 统一控制 scope、permission、grounding 和 trace。

项目价值

核心价值是让业务规则 KB 可以被服务和平台复用。

RTS 不只是降低检索成本。它把报文转换里的业务规则逻辑整理成可治理、可发布、可迁移的 KB，使规则可以进入查询、审计、迁移、影响分析、测试规划和 LLM/Agent 分析等业务场景。

业务逻辑可沉淀

规则不再停留在零散文档或代码里，而是进入结构化 KB。

跨平台可复用

不同服务、agent 或业务平台可以基于同一套 KB/projection 取得一致答案。

运行边界可控制

service 统一管理 release、scope、permission、grounding 和 trace。

LLM/Agent 分析有依据

内部 LLM 和外部 agent 基于 governed KB 做解释、影响分析和测试建议。

项目主线

从材料，到 KB，再到 runtime projection。

RTS 先把业务规则逻辑治理成 KB，再发布为 runtime projection；搜索、API、内部 LLM 和外部 agent 都基于这份运行包工作。

核心过程

RTS 建立的是“业务规则 KB 发布链路”。

规则先进入 KB，再发布成 projection，最后由服务、API、内部 LLM 和外部 agent 复用。

01

收集材料

业务文档、Java/XSLT、Excel mapping、lookup、样例和 review 结论。

02

整理成 KB

规则、lookup、helper、证据链、歧义、人工裁决和 signoff 被结构化保存。

03

发布投影

approved 内容被生成 release manifest、scope、object manifest、L2、dependency 等运行包。

04

服务消费

查询、API、搜索、内部 LLM 和外部 agent 都从同一份 projection 读取。

Service / LLM / Agent

RTS service 把 KB 能力开放给内部 LLM 和外部 agent。

内部 LLM 走 managed analysis；外部 agent 走 REST/MCP tool mode。两种模式都由 RTS service 控制 scope、grounding、permission 和 trace。

Managed Analysis

RTS 内部 LLM 在服务边界内基于 KB/projection 组织解释、影响分析、测试建议和调查路径。

Glossary · 术语解释

关键英文技术名词

页面以中文为主；英文术语保留在关键位置，是为了和代码、API、文档标题保持一致。下面给出阅读时最容易混淆的词。

RTS Rule Truth Source / Transformation Rule System

RTS 将分散的转换规则整理成可治理、可发布、可查询的业务规则 KB。

KB Knowledge Base，业务规则 KB

保存规则逻辑、来源证据、review 结论、歧义和 signoff 的完整治理层。

Rule Pack 规则包

按业务范围组织的一组 rule、lookup、helper，是整理、review 和发布的基本单位。

Runtime Projection 运行时投影

把已批准的 KB 内容发布成服务可读取的运行包，让查询、索引和后续消费有稳定输入。

L2 Runtime Object L2 运行时规则对象

runtime projection 中最重要的事实正文，保存服务可读的结构化规则逻辑。

Object Card 对象卡片

用于定位、消歧和召回规则对象；它帮助找到 L2，但不能替代 L2。

Release Manifest 发布清单

说明当前 projection 来自哪里、包含什么、是否可用，以及出现问题时如何回滚。

Scope 业务范围

通常由 channel、product、pack、domain 组成，用来防止相似规则被串用。

Dependency 规则依赖

描述 rule、lookup、helper 之间的关系，是解释影响面和测试范围的基础。
