RTS 是什么

面向银行报文转换场景的双核心规则真相服务

一句话定义：RTS 从多种 source（Excel、Java、XSLT、业务文档等）中整理 transformation rules，经过 AI-first review 和人工裁决，形成可治理、可追溯、可查询的规则真相，并以两种核心能力对外提供服务。

Core 1

真相源原子能力服务

稳定提供事实查询、对象读取、依赖遍历、scope 导航、trace 审计等结构化能力。

像一个精确的图书馆管理员 —— 你问什么，它精确地找到并交给你，同时记录谁看了什么。

Core 2

托管 LLM 分析服务

在同一 truth boundary 内提供规则解释、影响分析、测试规划、冲突简化等 AI 驱动的高价值分析。

像一个有专业知识的分析师 —— 它能解释规则、发现冲突、建议测试方案，但最终决定权在人。

RTS 不是什么

✗ 不是静态文档库
✗ 不是普通 RAG 问答
✗ 不是聊天机器人
✗ 不是 Agent SDK 包装

数据流全景

一条规则从"原始材料"到"被服务消费"的完整旅程：

层级职责关键约束
Source提供原始材料不直接进入运行时
KB形成和治理真相是 truth 的唯一 owner
Projection发布可服务运行包不可变，每次发布新版本
Service受控读取和追踪不拥有 truth，只拥有 access boundary

KB 设计理念

KB 是 RTS 的完整 governed truth graph（受治理真相图谱）

KB 管理的完整生命周期

三类核心对象

KB 中的规则真相由三类对象共同构成：

Rule

规则对象

描述具体的 transformation logic。

包含：inputs、logic pipeline、target path、dependencies、examples

查看代码示例
id: rule_fxd_ndf_primary_rate_source
status: demo_signoff
inputs:
  - fixing_currency1  (XPath)
  - fixing_currency2  (XPath)
  - cutoff_code       (XPath)
logic.pipeline:
  - read_xpath → fixing_currency1
  - read_xpath → fixing_currency2
  - read_xpath → cutoff_code
  - lookup_value → hedge_primary_source
  - lookup_value → hedge_primary_source_page
target: .../conf:primaryRateSource
dependencies: [lk_fxd_ndf_cutoff_by_pair_and_locode]
examples: {USD,TWD,TW} → {RTRS, TAIFX1}
Lookup

查找表对象

定义 key → value 映射，支持 fallback。

包含：key composition、return fields、fallback behavior

查看代码示例
id: lk_fxd_ndf_cutoff_by_pair_and_locode
source: "TraditionStella Cutoff" 映射表
logic:
  - concat(ccy1, ccy2) → forward_pair
  - compose_key("{pair}|{code}")
  - lookup(forward_key) → result
  - coalesce(forward, reverse)  ← fallback
output: cutoff_name, fixing_time,
  fixing_business_center,
  hedge_primary_source
examples:
  {USD,TWD,TW} → key="USDTWD|TW" → 命中
  {TWD,USD,TW} → 反向 fallback 命中
Helper

辅助逻辑对象

封装可复用的中间计算逻辑。

包含：inputs、logic pipeline、output fields

查看代码示例
id: hlp_fxd_ndf_fixing_quoted_currency_pair
inputs: source_currency1, source_currency2,
        source_quote_basis
logic:
  - concat(ccy1, ccy2) → source_pair
  - equals(quote_basis, source_pair) → is_inverse
  - branch(is_inverse):
      then: swap → Currency1PerCurrency2
      else: keep → Currency2PerCurrency1
outputs: target_currency1, target_currency2,
         target_quote_basis
examples:
  {USD,TWD,"USDTWD"} → inverse → {TWD,USD}

AI-first Governance

不是"AI 做决定"，而是"AI 尽可能完成机械工作，人工聚焦在最终裁决"：

AI 可以做

提取候选规则 · 结构校验 · 跨源验证 · 发现冲突和缺失 · 简化 reviewer 问题 · 解释依赖和影响

AI 不能做

发明缺失逻辑 · 静默解决冲突 · 覆盖人工裁决 · 未经签核发布 · 把候选当已批准真相

Pack 结构

Rule Pack 是 KB 中规则整理、治理、review 和 signoff 的基本组织单元

Pack 的两层结构

设计原则：主对象保持精简，治理信息分离。长 evidence、review 历史不混入 rule body。主对象层描述"做什么"，治理层解释"为什么可信"。

Pack 的 Scope

每个 pack 都有明确的 scope 定义。Scope 是 truth safety boundary：

source_system:   Tradition          ← 来源系统
target_system:   Stella             ← 目标系统
rule_domain:     cutoff-fixing      ← 规则领域
product_scope:   FXD.NDF            ← 产品范围
status:          demo_signoff       ← 当前状态

为什么 scope 如此重要？FXD 和 FXO 有相似字段名但不同逻辑。没有 scope 隔离，系统可能把 FXO 的规则当成 FXD 的答案返回 —— 这比"找不到"更危险。

Runtime Projection：运行时投影

把已批准的规则真相发布给运行时服务使用的产品化运行包

类比：如果 KB 是一本完整的法律全书（包含立法过程、辩论记录），那么 Runtime Projection 就是法院使用的现行有效法规汇编 —— 只包含已生效的条文，但保留了足够的结构让法官能准确引用。

Projection 解决的产品问题

问题Projection 如何解决
当前线上应该用哪个规则版本？active-release.json 指向当前版本 + rollback target
用户的问题属于哪个业务范围？scopes.jsonl 定义 channel/product/pack 边界
哪些规则已 released？object-manifest.jsonl 列出所有可服务对象
最终事实从哪里读取？l2/ 目录下的结构化 runtime objects
如何低噪声找到正确规则？navigation/ 提供 cards、aliases、confusables
规则之间有什么依赖？dependencies/ 记录 edges 和 field bindings
每次回答读了什么？traces/ 记录完整审计轨迹

Projection 与 KB 的关系

KB

完整 governed truth graph · 包含所有 evidence/review/冲突 · 是 truth 的唯一 owner · 修改规则必须回到这里

Runtime Projection

KB 发布出来的 approved truth 服务视图 · 裁剪了治理噪声 · 不可变 · 是 RTS Service 的唯一默认输入

不可逆向写入：任何人或 AI 都不能通过 query trace、agent memory 或搜索结果反向写回 truth。修改规则必须回到 KB 的 governance → signoff → release 流程。

L0 / L1 / L2 分层精度视图

同一个资源的不同精度视图 —— 不是目录层级，而是 precision views

真实示例：同一条规则的三层视图

以 rule_fxd_ndf_primary_rate_source 为例：

L0 极短摘要
{
  "uri": "rts://tradition-stella/fxd-ndf/rule/primary-rate-source",
  "type": "rule",
  "scope": "Tradition→Stella / FXD.NDF / cutoff-fixing",
  "summary": "Resolve hedge primary rate source from cutoff lookup by currency pair"
}

只有 4 个字段。足够判断"是否可能相关"，但不能回答事实问题。

L1 结构化卡片
{
  "uri": "rts://tradition-stella/fxd-ndf/rule/primary-rate-source",
  "type": "rule", "target_kind": "block",
  "target_path": ".../conf:primaryRateSource",
  "source_anchors": ["quotedCurrencyPair/currency1","currency2","fixingTime"],
  "logic_ops": ["read_xpath", "lookup_value"],
  "dependencies": {"lookups": ["lk_fxd_ndf_cutoff_by_pair_and_locode"]},
  "applicability": "FXD.NDF trades with nonDeliverableForward node",
  "confusables": ["rule_fxo_vanilla_primary_rate_source (wrong product)"],
  "status": "demo_signoff"
}

足够判断"确实是我要找的，不是 FXO 那条"。但不包含完整逻辑。

L2 完整结构化事实 ★
{
  "id": "rule_fxd_ndf_primary_rate_source",
  "status": "demo_signoff",
  "inputs": [
    {"name":"fixing_currency1","source_path":".../currency1/text()"},
    {"name":"fixing_currency2","source_path":".../currency2/text()"},
    {"name":"cutoff_code","source_path":".../hourMinuteTime/text()"}
  ],
  "logic": {"pipeline": [
    {"op":"read_xpath","out":"fixing_currency1"},
    {"op":"read_xpath","out":"fixing_currency2"},
    {"op":"read_xpath","out":"cutoff_code"},
    {"op":"lookup_value","lookup":"lk_fxd_ndf_cutoff...","return_field":"hedge_primary_source"},
    {"op":"lookup_value","lookup":"lk_fxd_ndf_cutoff...","return_field":"hedge_primary_source_page"}
  ]},
  "target": {"kind":"block","path":".../conf:primaryRateSource",
    "emits": [{"rateSource":"from_lookup:hedge_primary_source"},
              {"rateSourcePage":"from_lookup:hedge_primary_source_page"}]},
  "dependencies": {"lookups":["lk_fxd_ndf_cutoff_by_pair_and_locode"]},
  "examples": [{"input":["USD","TWD","TW"],"result":["RTRS","TAIFX1"]},
               {"input":["AUD","GBP","NY"],"result":["",""]}],
  "content_hash": "sha256:a3f7..."
}

这才是服务回答事实时必须读取的内容。

核心约束：如果只有 L0/L1 命中但没有 L2 对象，查询层应该拒答或降级。搜索命中、card、LLM summary 都不能替代 L2。

运行包的 11 个区域

Runtime Projection 像一个 release package，每类文件承担一个产品角色

1 active-release.json — 当前服务指针

回答：现在运行哪一版？回滚到哪一版？包含 active release id、rollback target。

2 release-manifest.json — 发布身份证

回答：这版从哪里来？能不能用？包含 release id、schema version、blocking issues count。没有有效 manifest 就没有可用 runtime truth。

3 scopes.jsonl — 业务范围注册表

防串线机制。Query resolver 必须先解析 scope 再执行召回。

4 object-manifest.jsonl — 对象目录

列出每个可服务对象的 stable URI、type、scope、content hash。确定性查找应优先从这里开始。

5 l2/ — 事实正文 ★ 最重要

保存服务可读的结构化规则事实：inputs、logic pipeline、target path、dependencies、examples、warnings、content hash。

搜索命中、card、HTML 展示、LLM summary 都不能替代 L2。

6 navigation/ — 查找和消歧

object-cards · l0-l1-views · aliases · confusables。让调用方不用知道精确 id 也能找到正确对象。

7 dependencies/ — 关系和影响

dependency-edges · field-bindings。影响分析、解释链和测试规划的基础。

8 governance/ — 可信度和风险

受权限控制的治理视图。普通查询不混入治理噪声；授权场景可展开。

9 caller-profiles.jsonl — 权限边界

定义谁能看哪些 pack、调用哪些入口。不同用户/pipeline/agent 不应天然拥有同样的权限。

10 index-artifacts/ — 检索加速

Projection 派生出的索引副本。帮助更快召回候选，但不能成为 final truth source。

11 traces/ — 审计复盘

记录 query plan、resolved scope、L2 read URIs、refusal reason、grounding map。可解释、可复盘、可审计。

真实用例

用 FXD.NDF Fixing 规则包中的真实对象来理解 KB 设计

用例 1：Block Rule — 门控规则

rule_fxd_ndf_fixing_block 决定 fixing 整个子树是否应该出现在目标报文中。

id: rule_fxd_ndf_fixing_block
logic:
  1. exists(/FpML/.../nonDeliverableForward) → 检查源报文是否有 NDF 节点
  2. emit_block(when: exists) → 如果有，渲染整个 fixing 块
target: .../conf:nonDeliverableSettlement/conf:fixing
children: [quotedCurrencyPair, fixingDate, primaryRateSource, secondaryRateSource, fixingTime]
dependencies.rules: [5 条子规则]
examples:
  ✓ NDF 存在 → block emitted
  ✗ NDF 不存在 → block not_emitted

用例 2：Helper — 可复用中间逻辑

hlp_fxd_ndf_fixing_quoted_currency_pair 封装货币对标准化逻辑，多条 rule 共享。

id: hlp_fxd_ndf_fixing_quoted_currency_pair
inputs: source_currency1, source_currency2, source_quote_basis
logic: "quoteBasis 等于拼接的货币对时，需要反转顺序"
  1. concat(currency1, currency2) → source_pair ("USDTWD")
  2. equals(quote_basis, source_pair) → is_inverse
  3. branch(is_inverse):
     then: swap currencies → Currency1PerCurrency2
     else: keep order → Currency2PerCurrency1
outputs: target_currency1, target_currency2, target_quote_basis
examples:
  {USD, TWD, "USDTWD"} → is_inverse=true → {TWD, USD, Currency1PerCurrency2}
  {USD, TWD, "Currency2PerCurrency1"} → is_inverse=false → {USD, TWD, Currency2PerCurrency1}

用例 3：Lookup — 共享查找表

lk_fxd_ndf_cutoff_by_pair_and_locode 被多条 rule 复用，核心特性是 reverse-pair fallback。

id: lk_fxd_ndf_cutoff_by_pair_and_locode
source: "TraditionStella Cutoff" Excel 映射表
logic: "先正向查，查不到再反向查"
  1. concat(ccy1, ccy2) → "USDTWD"
  2. compose_key("USDTWD|TW")
  3. lookup(forward_key) → forward_result
  4. concat(ccy2, ccy1) → "TWDUSD"
  5. compose_key("TWDUSD|TW")
  6. lookup(reverse_key) → reverse_result
  7. coalesce(forward, reverse)
output: cutoff_name, fixing_time, fixing_business_center, hedge_primary_source
examples:
  {USD,TWD,TW} → key="USDTWD|TW" → 正向命中 → {TPE,"11:00:00","TW TPE",RTRS,TAIFX1}
  {TWD,USD,TW} → 正向未命中 → 反向"USDTWD|TW"命中 → 同上
  {AUD,GBP,NY} → 正向命中但 hedge 为空 → {NY,"10:00:00","US NYC","",""}

用例 4：Rule 调用 Lookup

rule_fxd_ndf_primary_rate_source 展示 rule 如何调用 lookup 并绑定到目标字段。

id: rule_fxd_ndf_primary_rate_source
inputs: fixing_currency1, fixing_currency2, cutoff_code (from FpML XPath)
logic:
  1-3. read_xpath → 读取三个输入
  4. lookup_value(lk_fxd_ndf_cutoff..., "hedge_primary_source") → out
  5. lookup_value(lk_fxd_ndf_cutoff..., "hedge_primary_source_page") → out
target:
  .../conf:primaryRateSource/conf:rateSource       ← from_lookup
  .../conf:primaryRateSource/conf:rateSourcePage   ← from_lookup
dependencies: [lk_fxd_ndf_cutoff_by_pair_and_locode]

对象间依赖关系

查询演练

模拟一个真实查询从进入系统到返回答案的完整路径

场景：用户问 "Tradition 到 Stella 的 FXD.NDF 交易，USDTWD 的 primary rate source 是什么？"

1
Scope Resolution
解析 scope：Tradition→Stella | FXD.NDF | cutoff-fixing ✓ 明确
2
Permission Check
检查 caller 权限：allowed_packs: [fxd-ndf-cutoff-fixing-split] ✓
3
Object Discovery
通过 manifest 确定性查找：rule_fxd_ndf_primary_rate_source
排除 confusable：rule_fxo_vanilla_primary_rate_source (wrong product)
4
L2 Read
读取完整结构化规则：logic pipeline + target binding + examples
example: {USD,TWD,TW} → {RTRS, TAIFX1}
5
Dependency Load
加载 lookup L2：lk_fxd_ndf_cutoff_by_pair_and_locode (reverse-pair fallback)
6
Grounded Answer
Facts: USDTWD primary rate source = RTRS, page = TAIFX1
Citations: rule → lookup
Warnings: pack 状态为 demo_signoff
7
Trace Emit
记录 query_id、scope、resolved_objects、l2_reads、grounding、release

失败场景

Scope 不清

用户没指定产品 → 系统拒答，要求澄清

只有 L0/L1 命中，没有 L2

L2 不可读 → 降级："只能定位候选，不能给出规则事实"

依赖未发布

Rule 找到但 lookup 未 release → 降级："依赖未发布，无法提供完整事实"

Precedence 冲突

COMMON 与 product-specific 冲突未解决 → 拒答："需要人工裁决"

核心理念：Unknown is safer than wrong. 拒答或降级是正确的服务行为，不是失败。

核心原则速查

RTS KB 和 Runtime Projection 设计中最重要的原则

所有权边界

原则含义
KB owns truthKB 是真相的唯一拥有者，修改必须回到 KB
Release owns admissibilityRelease 决定哪些对象可以进入运行时
Projection owns shapeProjection 决定索引层可以读取的结构
L2 owns rule contentL2 是事实来源，回答必须回到这里
LLM answer never owns truthLLM 输出是分析和表达，不是新的真相

安全原则

原则含义
Unknown is safer than wrong证据不足时拒答或降级
Scope first, content later先收窄范围再读取内容
Context must be precise精确上下文比更多上下文更安全
Evidence outranks narrative证据优先于叙述

AI 输出的五类区分

Facts

能引用 L2/dependency/evidence

Inferences

基于 facts 的分析推断

Unknowns

证据不足或存在冲突

Candidates

AI 生成、等待人确认

Human Decisions

reviewer 已裁决的结果

判断标准

设计选择时的决策规则

"如何读取和证明 RTS truth" → 放在 RTS service
"如何理解外部场景输入" → 交给 LLM harness 或外部 agent
"如何把结果写成报告" → 交给 LLM
"这个 claim 是否可以作为事实" → 必须回到 L2 / dependency / governance
"该输出能否驱动自动化动作" → 只有 deterministic contract 或 recorded human decision 可以

总结：RTS 的核心不是"把规则文档放进 AI 问答"，而是建设一个以多源 evidence、AI-first governance、人工最终裁决和权限化服务发布为基础的规则真相服务。

KB 负责形成和治理真相，Runtime Projection 负责把 approved truth 安全地交付给服务层，L0/L1/L2 分层确保检索精确且高效，而 AI 在这些明确边界内提供高价值的分析和表达能力。
