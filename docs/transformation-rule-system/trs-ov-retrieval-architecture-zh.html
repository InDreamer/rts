真相治理与低幻觉检索架构图

这张图把当前思想压成一条清晰主线：先在治理侧定义真相，再做单向只读投影，然后通过 L0 召回 → L1 重排 → L2 最终加载 把最小必要上下文送进 agent 运行时。运行时可以使用真相，但不能反过来改写真相。

图的阅读顺序
1. 先看四层职责
2. 再看中间的检索链
3. 最后看红色禁止路径
主视角：分层责任 + 检索链路 核心纪律：先认证真相，再供运行时读取

架构主旨

  • • TRS 负责定义“什么是真的”，OV 负责让 agent 更容易找到并读取真相。
  • • 投影层只接受已 signoff 的主对象，不把治理过程原样搬进运行时。
  • • 运行时层只能使用上下文，不允许把使用痕迹反向升级为真相。

检索链解释

  • • 召回读 L0：先保覆盖，避免漏掉正确 pack 或对象。
  • • 重排读 L1：利用结构摘要收紧范围，把相似但错误的候选剔除。
  • • 最终加载读 L2：只拉入已确认命中的原始 YAML，并顺着依赖去拿 lookup / helper。

边界纪律

  • • approved only：draft、review-ready 不进入第一阶段 runtime surface。
  • • evidence / review / reports 留在治理侧，因为它们回答“为何成立”，不是运行时首层答案。
  • • 优先用地址、依赖和层级导航控噪，而不是一开始就做全文混合检索。
