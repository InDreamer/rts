如何使用

使用 RTS 的前提，是先有一份 runtime projection。

稳定入口是 projection store：它定义当前 release、可用 scope、可查询对象和事实正文位置。

1选择 release

active-release 指向当前服务应该使用的规则版本。

2解析 scope

channel/product/pack/domain 防止相似规则串用。

3读取 L2

事实回答回到结构化规则正文和依赖关系。

4留下 trace

记录这次查询读了什么，方便复盘和审计。

01

本地 demo 启动

当前 demo 使用 sample runtime projection store。它展示的是”服务读取 projection”的流程，而非直接读取 KB 草稿。

RTS_STORE_ROOT="$PWD/sample-projection/runtime-store" \
JAVA_HOME=$(/usr/libexec/java_home -v 17) \
mvn spring-boot:run
02

先验证服务面

先查看可用工具或 API，再使用当前 active release 的 sample scope 查询。

curl -s http://localhost:8080/mcp/tools | jq
03

确认变更可发布

如果修改了文档、projection 或服务代码，提交前至少要跑测试和格式检查。

JAVA_HOME=$(/usr/libexec/java_home -v 17) mvn verify
git diff --check
git status --short

Glossary · 术语解释

使用时会遇到的英文词

这些词会出现在 projection 包、API 请求、响应和 trace 中。

RTS Rule Truth Source / Transformation Rule System

RTS 将分散的转换规则整理成可治理、可发布、可查询的业务规则 KB。

KB Knowledge Base，业务规则 KB

保存规则逻辑、来源证据、review 结论、歧义和 signoff 的完整治理层。

Rule Pack 规则包

按业务范围组织的一组 rule、lookup、helper，是整理、review 和发布的基本单位。

Runtime Projection 运行时投影

把已批准的 KB 内容发布成服务可读取的运行包，让查询、索引和后续消费有稳定输入。

L2 Runtime Object L2 运行时规则对象

runtime projection 中最重要的事实正文，保存服务可读的结构化规则逻辑。

Object Card 对象卡片

用于定位、消歧和召回规则对象；它帮助找到 L2，但不能替代 L2。

Release Manifest 发布清单

说明当前 projection 来自哪里、包含什么、是否可用，以及出现问题时如何回滚。

Scope 业务范围

通常由 channel、product、pack、domain 组成，用来防止相似规则被串用。

Dependency 规则依赖

描述 rule、lookup、helper 之间的关系，是解释影响面和测试范围的基础。

Governance View 治理视图

在权限允许时展示 evidence、review、open question、risk 和 signoff 信息。

介绍和使用时容易误解的点

  • Projection 不是摘要：它保留 L2 结构化规则语义。
  • 搜索结果不是事实：card、L0/L1、index document 负责定位对象，事实在 L2。
  • LLM/Agent 是消费层：分析和表达基于 KB 与运行包，而非独立生成。
