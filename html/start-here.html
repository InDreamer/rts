RTS 项目介绍

项目概览

RTS 将分散的转换规则整理成可治理的 KB，并发布为 runtime projection，支撑查询、审计和后续分析。

项目概览

RTS 先解决“规则散落且难以证明”的问题。

在报文转换场景里，同一条规则可能散在业务文档、Java/XSLT、Excel mapping、lookup table、样例和 review 记录里。RTS 的第一目标不是做聊天机器人，而是把这些材料整理成可治理的 KB，再发布成 runtime projection，供运行时服务读取。

现状 规则分散，靠人翻找和解释
建设 KB 沉淀规则、证据和 review
交付 Runtime projection 支撑服务查询
01

KB 是规则知识的治理层。

它保存 rule、lookup、helper、source、evidence、review、ambiguity、adjudication 和 signoff。这里关心的是规则为什么可信。

02

Runtime projection 是运行时交付层。

它把 approved truth 生成 release、scope、object manifest、L2 runtime object、dependency 和 governance summary，供服务稳定读取。

03

API/AI 是消费层，不是事实源。

查询、索引、影响分析和 AI 辅助都应该基于 projection 读取规则，而不是直接猜测、搜索草稿或读取未批准材料。

核心表述

五句话概括。

  1. RTS 不是把规则文档简单丢进搜索，而是整理一套可治理的规则知识库。
  2. KB 保存完整治理上下文：规则逻辑、来源证据、review、歧义、裁决和 signoff。
  3. Runtime projection 把已批准内容发布成运行时服务可以读取的稳定包。
  4. 服务回答必须能回到 L2 规则对象、依赖关系、授权治理视图或 trace。
  5. AI 可以帮助解释和分析，但它消费的是这套已治理材料，不替代规则事实。

Glossary · 术语解释

核心术语

这些词会在介绍、文档和接口里反复出现；英文保留是为了和真实文件名、字段名和接口一致。

RTS Rule Truth Source / Transformation Rule System

RTS 将分散的转换规则整理成可治理、可发布、可查询的规则知识基础。

KB Knowledge Base，规则知识库

保存规则逻辑、来源证据、review 结论、歧义和 signoff 的完整治理层。

Rule Pack 规则包

按业务范围组织的一组 rule、lookup、helper，是整理、review 和发布的基本单位。

Runtime Projection 运行时投影

把已批准的 KB 内容发布成服务可读取的运行包，让查询、索引和后续消费有稳定输入。

L2 Runtime Object L2 运行时规则对象

runtime projection 中最重要的事实正文，保存服务可读的结构化规则逻辑。

Object Card 对象卡片

用于定位、消歧和召回规则对象；它帮助找到 L2，但不能替代 L2。

Release Manifest 发布清单

说明当前 projection 来自哪里、包含什么、是否可用，以及出现问题时如何回滚。

Scope 业务范围

通常由 channel、product、pack、domain 组成，用来防止相似规则被串用。

视角

不同场景的关注点。

价值是把规则资产沉淀下来，而不是只做一个问答入口。

可以关注：规则来源能否追溯、发布版本是否清楚、后续是否能支撑查询、审计、影响分析和自动化。

关键结构是 KB 到 runtime projection 的边界。

可以关注：manifest、scope、object manifest、L2 runtime object、object card、dependency、governance summary 和 trace。

AI 辅助不是第一叙事，但它会受益于这套结构。

当规则被整理成 projection 后，AI 可以更稳定地做解释、影响候选、测试建议和 review 辅助。

治理重点是证据、歧义和裁决能不能留下来。

KB 里要记录 source、evidence、review、open question、adjudication 和 signoff；projection 只发布运行时需要的视图。
